## Key decisions / known limitations (carry forward)
- **JE-only Credit Note & Interest** (no migration). Deferred: a credit-note *document* model (`doc_type` on `acct_invoices`), dedicated `credit_note`/`interest` journal `source` enum values, and per-customer **sub-ledger allocation**.
- **Sub-ledger vs GL gap (biggest accounting limitation):** payments, credit notes, and interest all post to *total* GL AR (1100), not allocated to a specific customer/invoice ‚Äî there's **no allocations table**. AR reports sum *open invoices*, so the sub-ledger and GL AR can diverge. Flagged in the relevant UIs.
- **GL posting rules:** all writes go through `recordJournalEntry`; AR (1100) is a control account ‚Äî posts must use a non-`manual` `source` (we use `'invoice'`).

## Working conventions (so tomorrow matches today)
- **One PR per logical unit, branched off `main`** ‚Äî avoid stacking (a stacked PR auto-closed when its base branch was deleted; recovered via rebase). Squash-merge; only `--delete-branch` when nothing is stacked.
